# Trusted WLAN Access Gateway (TWAG)

Минимальный Proof-of-Concept для шлюза **Trusted WLAN Access Gateway (TWAG)**, готового к интеграции с ядром мобильной связи (MNO). Данный проект представляет собой асинхронный C++ сервис, выступающий мостом между доверенными Wi-Fi сетями (AP/WLC) и ядром мобильного оператора (EPC).

## Функционал проекта

Проект реализует сквозной (end-to-end) асинхронный поток аутентификации абонента (UE) по протоколу EAP-AKA/SIM и маршрутизации пользовательского трафика (Data Plane) на уровне ядра ОС Linux.

### 1. TWAN (RADIUS) Фронтенд & Контроль перегрузок (Congestion Control):
- **Асинхронный UDP-сервер** (порт 1812) для приема запросов `Access-Request` и `Accounting-Request` от Wi-Fi точек доступа (AP).
- **Парсинг RADIUS TLV:** Извлечение MAC-адреса устройства (`Calling-Station-Id`) и нагрузки `EAP-Message`.
- **EAP Parser:** Глубокий анализ пакетов `EAP-Identity` для извлечения **IMSI** абонента, что необходимо для создания сессии в ядре оператора.
- **Поддержка Multi-Round EAP:** Генерация ответов `Access-Challenge` для поддержки сложных протоколов аутентификации мобильных сетей (EAP-AKA/SIM).
- **Rate Limiting:** Динамическое ограничение количества активных сессий (настраивается в JSON). При превышении лимита новые запросы отклоняются (`Access-Reject`).

### 2. STa Интерфейс (Diameter):
- Интеграция с демоном `freeDiameter` для связи с AAA-сервером (HSS/AUSF).
- Инкапсуляция RADIUS EAP-нагрузки в Diameter `EAP-Payload` (запрос DER).
- Асинхронные коллбэки (`dea_cb`) для обработки ответов Diameter EAP Answer (DEA). Парсинг профиля абонента (APN, PDN-Type, AMBR).
- **Динамический DNS (S-NAPTR):** Разрешение IP-адреса P-GW на основе FQDN APN через DNS-запросы `ns_t_naptr` (с использованием `libresolv`).

### 3. S2a Интерфейс (GTPv2-C - Control Plane):
- Интеграция библиотеки `libgtpv2c` для сигнального взаимодействия с пакетным шлюзом (PGW).
- Отправка `Create Session Request`, включающего извлеченный **IMSI** (в формате BCD), а также динамически сгенерированные локальные F-TEID (с определением локального IP).
- Обработка `Delete Session Request` с использованием строгой сериализации структур `del_sess_req_t`.
- Парсинг ответов `Create Session Response` с извлечением **PGW F-TEID**, **UE IP (PAA)** и **QCI (Quality of Service Class Identifier)**.

### 4. S2a Интерфейс (GTP-U - User Plane, EoGRE & QoS):
- **EoGRE туннелирование:** Автоматическое создание L2 GRE туннелей (`gretap0`) до WLC через вызовы ядра Linux.
- Динамическое создание сетевых интерфейсов `gtp0` с помощью `libgtpnl` (Netlink).
- **L3 IP Routing:** Установка маршрутов в `gtp0`.
- **DHCP Сервер:** Встроенный DHCP-сервер, жестко привязанный к `gretap0` (`SO_BINDTODEVICE`), выдающий IP-адреса, полученные от P-GW, абонентам.
- **QoS (DSCP Marking):** Динамическое управление правилами `iptables` в таблице `mangle` для маппинга EPC QCI в DSCP (например, QCI 1 -> EF, QCI 5 -> CS3) для нисходящего трафика.
- Сбор статистики по сессиям и периодическое логирование.

### 5. Архитектура и Инструменты:
- Современный стандарт **C++17**, сборка через **CMake**.
- **epoll** для полностью неблокирующей обработки пакетов.
- **yajson** для парсинга конфигурационного файла `twag_config.json`.
- **GoogleTest** для интеграционного тестирования стейт-машины и кодировщиков GTP.

## Тестирование и Запуск (E2E Test Suite)

Проект включает мощный сквозной (End-to-End) тестовый набор, написанный на Python, который эмулирует все узлы мобильной сети и Wi-Fi контроллера.
Скрипт `test_mno.sh` автоматически компилирует проект, загружает модуль ядра Linux `gtp`, и поднимает:
1. `mock_dns.py` - Эмулятор DNS для тестирования S-NAPTR.
2. `mock_pgw.py` - Эмулятор P-GW (GTPv2-C).
3. `test_client.py` - Клиент для стресс-тестирования (RADIUS Attach, Detach, Congestion Control).

```bash
# Тестирование требует sudo (для поднятия интерфейсов gretap0, gtp0 и iptables)
sudo ./test_mno.sh
```

В ходе работы скрипта вы увидите:
1. Прохождение юнит-тестов (C++ GTest).
2. Запуск TWAG и обработка S-NAPTR запроса к Mock DNS.
3. Отправка GTPv2-C запросов к Mock P-GW и парсинг ответов (извлечение QCI и IP).
4. Успешное создание интерфейсов `gretap0` и `gtp0`.
5. Применение QoS DSCP правил через `iptables`.
6. Срабатывание системы Rate Limiting при попытке открыть больше сессий, чем разрешено в конфигурации.
7. Корректную очистку всех интерфейсов, маршрутов и правил QoS при получении `Accounting-Request (Stop)`.
